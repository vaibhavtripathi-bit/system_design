What This Document Is
This is the companion document to the Android System Design L6 prep kit. Where the first document focused on Android-specific infrastructure problems (caching, sync, pagination), this one targets a different dimension of the L6 interview: pure object-oriented system design. Problems like designing a lift system, a parking lot, a vending machine, or a chess game test whether a candidate can model a complex real-world domain cleanly, reason about edge cases upfront, apply the right design patterns, and structure code so that new requirements can be absorbed without rewrites. Same rules as the first doc apply: one Android module per problem, multiple approaches per problem in separate packages, all verified by unit tests only ‚Äî no UI.
The L6 Lens ‚Äî SOLID at Every Level

Every problem in this set should be evaluated through the SOLID lens. An L6 candidate doesn't just implement ‚Äî they proactively identify where each principle is stressed, and design accordingly.

S
Single Responsibility
Each class owns exactly one reason to change
O

Extend without modifying existing code

Extensibility Framework ‚Äî Required Per Problem

For each problem, the solution must explicitly address the following extensibility axes. This is what L6 looks like versus L5 ‚Äî not just "it works," but "here's where the seams are."

Critical Edge Cases

  • What happens at capacity limits?
  • What if a dependency fails mid-operation?
  • Concurrent access to shared state
  • Invalid or out-of-order inputs
  • Partial failure scenarios

Future Feature Hooks

  • Where would a new type be added?
  • Which interfaces absorb new behaviour?
  • What would require a breaking change?
  • Where is the Open/Closed boundary?

State Machine Design

  • Is there an explicit state machine?
  • Are invalid transitions guarded?
  • Can state be persisted and restored?
  • Is the state machine testable in isolation?

Observer / Event Hooks

  • Who needs to react to state changes?
  • Push vs pull notification model
  • How are listeners decoupled from source?
  • Lifecycle of listener registration

01
Lift / Elevator System
Critical
Problem Scope
Design a multi-lift building system. Handle floor requests (inside cabin and outside buttons), dispatch algorithm, door mechanics, overweight sensor, emergency stop, and maintenance mode. Model the full lifecycle of a single trip.
Critical Cases to Solve
  • Multiple simultaneous requests ‚Äî dispatch strategy
  • Lift at capacity ‚Äî reject vs queue
  • Door blocked / sensor fault mid-close
  • Power failure mid-floor ‚Äî safe landing protocol
  • Adding a new dispatch algorithm without touching Lift
  • VIP floor access (future extensibility hook)
State Machine (IDLE‚ÜíMOVING‚ÜíSTOPPED‚ÜíDOORS_OPEN) Strategy Pattern (dispatch algorithms: FCFS, SCAN, LOOK) Command Pattern (floor requests as queued commands) Observer (door/weight sensors notify controller)
02
Parking Lot System
Critical
Problem Scope
Multi-floor parking lot with different spot types (compact, large, handicapped, EV), multiple entry/exit gates, ticket generation, fee calculation, and spot assignment strategy.
Critical Cases to Solve
  • Lot full ‚Äî all entry gates reject simultaneously
  • Ticket lost ‚Äî fallback pricing policy
  • EV spot requested but none free ‚Äî degrade to regular
  • Multi-vehicle type extending without changing lot logic
  • Dynamic pricing (future hook)
  • Concurrent entry/exit race condition
Strategy (spot assignment: nearest, by type, by floor) Factory Pattern (vehicle + spot type creation) Decorator (pricing: base + time + peak + discount) Template Method (entry/exit flow with overridable steps)
03
Vending Machine
Critical
Problem Scope
A vending machine with multiple product slots, coin/note payment, change dispensing, refill mode, and admin override. Handle partial payment, expired products, and stuck dispenser scenarios.
Critical Cases to Solve
  • Exact change not available ‚Äî refuse or notify
  • Product stuck in dispenser ‚Äî compensation flow
  • Payment inserted for out-of-stock item
  • Admin refill mid-transaction
  • New payment type (card/QR) added without core changes
State Machine (IDLE‚ÜíHAS_MONEY‚ÜíDISPENSING‚ÜíCHANGE) Strategy (payment processor interface) Chain of Responsibility (coin validation pipeline) Command Pattern (transaction as reversible commands)

High

Chess Game Engine
Critical
Problem Scope
Board representation, piece movement rules, check/checkmate/stalemate detection, move validation, castling, en passant, pawn promotion. Support for game replay and undo.

Claude Code Prompt ‚Äî OO System Design

Set up an OOSystemDesign/ directory in this Android project. For each problem below, create a separate Android module. These problems have no dependency on Android internals ‚Äî all logic should be pure Kotlin where possible.

Structure Rules

  • One module per problem, kebab-case naming (e.g. elevator-system/, parking-lot/)
  • One package per approach inside each module: approach_01_state_machine/, approach_02_strategy_pattern/ etc.
  • No UI required ‚Äî empty Activity only if Android module structure demands it
  • All logic verified via Unit Tests only ‚Äî JUnit, MockK, Coroutines Test as needed

Per-Problem Requirements

  • KDoc block per approach: pattern used, trade-offs, when to prefer it
  • Explicit modelling of critical edge cases (listed per problem below) ‚Äî each must have a unit test
  • Each problem must document: where a new type would be added, and what would require a breaking change
  • Minimum 2‚Äì3 distinct approaches per problem using different design patterns or architectural styles

Approach Variation Axes

  • Behavioural patterns: Strategy, Command, Observer, Chain of Responsibility, State Machine, Template Method, Visitor, Memento
  • Structural patterns: Decorator, Composite, Proxy, Facade, Adapter, Bridge
  • Creational patterns: Factory Method, Abstract Factory, Builder, Prototype
  • Architecture: Push vs Pull, Event-driven vs Request-response, Eager vs Lazy init
