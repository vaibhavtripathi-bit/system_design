Objective

This project is a hands-on Android system design study kit targeting the Google Staff Engineer (L6) interview. The goal is not to build production apps, but to deeply explore design trade-offs by solving each system design problem using multiple architectural patterns and approaches — all verified through unit tests. Every solution should be accompanied by clear reasoning about when that approach makes sense, what it trades off, and how it compares to the alternatives within the same module. This mirrors exactly what an L6 interview expects: breadth of knowledge, depth of reasoning, and the ability to navigate trade-offs confidently.

(one module per problem)

Module Rules

  • One Android module per problem
  • Module named clearly (kebab-case)
  • Empty Activity is acceptable — no UI needed
  • Each module is self-contained and independently runnable

Package Rules

  • One package per approach inside each module
  • Named approach_01_xxx/, approach_02_xxx/ etc.
  • Package name should reflect the pattern/approach used
  • Shared utilities may live in a common/ package

Documentation Rules

  • Each approach must have a top-level KDoc comment block
  • Document: pattern used, trade-offs, when to prefer this approach
  • Contrast explicitly with the other approaches in the same module
  • No external README files required — inline docs are sufficient

Testing Rules

  • All solutions verified via Unit Tests only — no UI tests
  • Use JUnit4/5, MockK, Kotlin Coroutines Test as needed
  • Tests must cover core behavior AND edge cases
  • Test class per approach, clearly named
Approach Philosophy

Each problem must be solved using multiple distinct approaches. Not every approach needs to be optimal — the goal is to demonstrate deep reasoning about trade-offs and articulate when each design makes sense. Approaches can vary along the following axes:

Design Pattern
Strategy · Command · Observer · Decorator · Chain of Responsibility · State Machine · Builder · Proxy · Template Method · Facade · Composite
Architectural Style
Pull vs Push · Reactive vs Imperative · Eager vs Lazy · Centralized vs Distributed · Event-driven vs Request-response
Concurrency Model
Coroutines + Flow · SharedFlow / StateFlow · Thread pools · Actors · Lock-free structures · Mutex vs CAS
Trade-off Axis
Memory vs Speed · Simplicity vs Extensibility · Consistency vs Availability · Latency vs Throughput · Strong vs Eventual consistency
20 System Design Problems
📡 Networking & Data Layer
#01
Image Loading Library
Memory + disk cache layering, request deduplication, cancellation, transformation pipeline. Similar scope to Glide/Coil internals.
#02
HTTP Client with Retry & Circuit Breaker
Exponential backoff, failure thresholds, half-open state, bulkhead isolation, timeout policies.
#03
Offline-First Sync Engine
Conflict resolution strategies (last-write-wins, CRDT-inspired, server-wins), dirty state tracking, sync queue management, retry scheduling.
#04
Pagination Engine
Page-based vs cursor-based, prefetching, boundary callbacks, error recovery, bidirectional loading.
💾 Storage & Caching
#05
LRU Cache
Thread-safe, memory-bounded, pluggable eviction policies (LRU, LFU, ARC, TinyLFU). Benchmark comparison across policies.
#06
Key-Value Store
In-memory vs persisted, write-ahead log (WAL), compaction strategies, snapshot isolation, ACID-lite guarantees.
#07
File Download Manager
Resumable downloads, priority queue (heap-based vs ordered), concurrency control, progress tracking, partial content handling.
🔔 Messaging & Events
#08
In-App Event & State Broadcasting System
SharedFlow vs StateFlow vs Channel, lifecycle-aware consumption, backpressure strategies, decoupled component communication. Highlight why EventBus is superseded by coroutine-based approaches.
#09
Push Notification System
Token lifecycle management, deduplication, local scheduling fallback, silent vs visible notification routing, batching.
#10
Real-Time Messaging System
WebSocket-based architecture, message ordering guarantees, delivery receipts, reconnection strategy with exponential backoff, presence detection.
⚙️ Concurrency & Scheduling
#11
Task Scheduler & Job Queue
Priority queues, dependency graphs (DAG-based chaining), retry policies with backoff, cancellation propagation, deadline scheduling.
#12
Rate Limiter
Token bucket vs leaky bucket vs sliding window log vs fixed window counter. Distributed vs local enforcement. Coroutine-safe implementations.
#13
Background Task System
Constraint-based execution (network, battery, storage), task chaining, persistence across restarts, WorkManager-inspired design.
📊 Observability & Reliability
#14
Analytics & Event Logging Pipeline
Batching strategies, flush triggers (time-based, size-based, lifecycle-based), ordering guarantees, loss tolerance, retry with deduplication.
#15
Crash & ANR Reporter
Signal and uncaught exception handling, breadcrumb trail construction, offline buffering, deduplication, symbolication pipeline.
#16
Feature Flag System
Remote config pull vs push, local override, gradual rollout (percentage bucketing), kill switch, experiment assignment, cache invalidation.
🧩 App Architecture
#17
Dependency Injection Framework
Graph resolution, scoping (singleton / component / factory), lazy vs eager initialization, circular dependency detection, simplified Dagger/Hilt internals.
#18
Navigation & Back Stack Manager
Deep link handling, state restoration, multi-stack (bottom nav), transition orchestration, result passing between destinations.
#19
Plugin & Dynamic Module System
Inter-module communication contracts, dynamic feature loading, isolation boundaries, versioning, dependency resolution between plugins.
🔒 Security & Auth
#20
Token Manager & Auth System
Access + refresh token lifecycle, concurrent request queuing during token refresh, secure storage (Keystore-backed), session expiry, multi-account support.

Recommended Tech Stack

Language & Runtime

  • Kotlin (primary) — idiomatic, coroutine-native
  • Kotlin Coroutines + Flow for concurrency
  • StateFlow / SharedFlow / Channel for reactive streams

Testing

  • JUnit 4 / JUnit 5
  • MockK for mocking Kotlin classes
  • kotlinx-coroutines-test for coroutine testing
  • Turbine for Flow testing

Build

  • Gradle with Kotlin DSL
  • One settings.gradle including all modules
  • Shared version catalog (libs.versions.toml)

No UI Dependencies

  • No ViewModel, LiveData, Compose, or XML layouts required
  • Avoid Android framework classes where possible — prefer pure Kotlin
  • Empty Activity only where Android module structure demands it
